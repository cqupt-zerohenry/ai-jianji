## 5.3 智能检测模块的关键设计说明

智能检测模块是本系统中最核心的组成部分之一，其主要任务是从比赛视频中识别进球、射门、扑救、犯规、越位、换人、黄牌、红牌、点球和 VAR 等关键事件，并将识别结果转换成后续自动剪辑能够直接使用的结构化数据。考虑到足球比赛视频本身具有镜头变化快、事件边界不清晰、单一模态信息不完整等特点，系统没有采用单链路检测方式，而是设计了由视觉检测链路和音频语义链路组成的双链路结构。

其中，视觉检测链路主要负责从比赛画面中提取事件线索，更适合识别进球、射门、扑救、角球等具有明显视觉特征的事件；音频语义链路则主要利用解说文本和现场语音信息进行语义判断，更适合补充识别犯规、越位、VAR 和判罚类事件。两条链路在同时成功运行时可以对结果进行融合，在其中一条链路不可用时也可以单独输出结果，从而提高系统在复杂场景下的可用性和稳定性。

---

## 5.3.1 视觉检测链路设计

视觉检测链路以比赛视频画面为主要输入，调用 DashScope 提供的多模态模型 `qwen3-vl-flash` 完成事件识别。该链路的核心思想是：先利用模型对整段视频进行粗粒度理解，再在长视频场景下通过时间窗口切分和关键帧抽样方式提高覆盖率，从而减少整段视频直接推理时的遗漏问题。

在具体实现中，系统首先读取视频文件并获取总时长，然后将整段视频直接提交给多模态模型进行一次整体分析。如果整体分析成功，则将模型返回的事件结果作为第一轮候选输出；如果整体分析失败，或者视频长度较大导致单次分析覆盖不足，则进一步把视频划分为多个时间窗口，并在每个窗口内抽取若干关键帧，再将这些关键帧连同统一提示词输入模型进行细粒度事件检测。最后，系统对整段视频分析结果与分窗分析结果进行合并、去重和规则过滤，形成视觉链路的最终事件集合。

视觉链路中使用的提示词如下所示。

### 视觉链路主提示词

```text
你是一名专业的足球比赛分析助手。请分析输入的比赛视频，识别其中的关键事件，并输出结构化 JSON 结果。

需要识别的事件类型包括：
- GOAL
- SHOT_ON_TARGET
- SHOT_BLOCKED
- SAVE
- CORNER_KICK
- FREE_KICK
- OFFSIDE
- FOUL
- SUBSTITUTION
- YELLOW_CARD
- RED_CARD
- PENALTY
- VAR
- HIGHLIGHT

进球判断规则如下：
1. 以画面证据为主，只有在球明显越过门线，或者存在强烈进球视觉证据时，才输出 GOAL；
2. 如果画面中出现越位、犯规、手球、进攻犯规或进球无效等迹象，则不能直接判为 GOAL；
3. 不能仅凭比分板变化或庆祝画面就认定为 GOAL；
4. 同一进攻回合不能重复输出多个 GOAL；
5. 如果射门被扑救、被封堵、射偏、击中门柱或被判无效，则不能输出 GOAL。

请仅返回 JSON，格式如下：
{
  "events": [
    {
      "event_type": "GOAL",
      "timestamp_seconds": 234.5,
      "confidence": 0.95,
      "description": "Header goal from corner kick"
    }
  ],
  "video_duration": 5400.0
}
```

### 分窗关键帧检测提示词

```text
你将收到某一时间窗口内抽取的若干关键帧，每一帧都带有绝对时间戳。请仅依据这些关键帧识别该时间窗口内可能发生的足球事件，并输出 JSON 结果。

要求如下：
1. 如果证据不足，不要凭空生成事件；
2. 只在视觉证据明确支持的情况下输出 GOAL；
3. 如果更像是射门但无法确认进球，应优先输出 SHOT_ON_TARGET、SHOT_BLOCKED 或 SAVE；
4. 对角球、任意球、越位、犯规、换人等事件，只有在裁判动作、球场画面或上下文足够清楚时才输出；
5. 避免对相近时间点的同一事件重复输出。

请仅返回 JSON，格式如下：
{
  "events": [
    {
      "event_type": "SHOT_ON_TARGET",
      "timestamp_seconds": 102.8,
      "confidence": 0.81,
      "description": "Shot on target from inside the box"
    }
  ],
  "video_duration": 5400.0
}
```

### 视觉链路提示词设计说明

视觉链路提示词的设计重点主要体现在以下几个方面。

第一，提示词中首先明确给出了事件分类集合，这样做的目的在于避免模型输出与系统定义不一致的自由标签。例如，如果不提前限定事件类型，模型可能会返回“dangerous attack”“crowd celebration”等不在系统 schema 中的标签，从而增加后续程序解析难度。

第二，提示词对 GOAL 的判断做了重点约束。这是因为在足球视频中，进球事件往往最重要，但同时也最容易被误判。比如，射门后的庆祝画面、比分字幕变化或者观众席镜头，都可能让模型过早给出 GOAL 判断。因此，提示词要求模型优先依据门线和进球动作本身做判断，而不是只看外围线索。

第三，提示词强制模型输出 JSON。这样做的目的不是单纯为了格式统一，而是为了让模型输出可以直接进入程序解析、事件落库和后续剪辑规划环节，减少人工清洗文本的步骤。

## 5.3.2 音频语义链路设计

相比视觉链路，音频语义链路并不直接分析视频画面，而是先从比赛视频中提取音频，再通过语音转写和文本语义抽取两个步骤识别关键事件。当前系统中，该链路同样基于阿里模型体系：首先使用 `qwen3-asr-flash` 完成语音转写，再使用 `qwen-plus` 对转写文本进行结构化事件抽取。

这种设计的原因在于，足球比赛中有不少事件并不完全依赖视觉画面判断。例如，解说员说出的“这球进了”“裁判吹停了比赛”“越位在先”“点球判罚”等内容，往往能够为事件识别提供直接语义线索。尤其在镜头切换频繁、画面证据不充分时，音频链路能够对视觉链路形成有效补充。

音频语义链路的执行过程可以分为以下几个步骤：

1. 使用 FFmpeg 从视频中抽取音频文件；
2. 调用 `qwen3-asr-flash` 对音频进行转写；
3. 将转写结果整理为带时间戳的文本序列；
4. 将时间戳文本送入 `qwen-plus`；
5. 由文本模型按固定 JSON 结构抽取事件；
6. 将结果转换成与视觉链路一致的事件对象格式。

### 音频转写阶段提示词

```text
请忠实转写输入音频内容，尽量保留自然的句子边界，不要补充原音频中不存在的信息。
```

这一阶段的提示词相对简单，因为这一阶段的目标不是事件识别，而是尽可能得到稳定的语音文本结果。其重点在于控制转写结果不要偏离原始语音内容，为后续事件抽取提供可靠输入。

### 音频语义事件抽取提示词

```text
你是一名足球比赛分析助手。下面给出的是带时间戳的比赛解说文本，请根据文本内容识别比赛中的关键事件，并仅输出 JSON。

需要识别的事件类型包括：
- GOAL
- SHOT_ON_TARGET
- SHOT_BLOCKED
- SAVE
- CORNER_KICK
- FREE_KICK
- OFFSIDE
- FOUL
- SUBSTITUTION
- YELLOW_CARD
- RED_CARD
- PENALTY
- VAR
- HIGHLIGHT

识别规则如下：
1. 请结合解说文本中的关键词和上下文语义判断事件类型；
2. 如果文本明确表示“进球”“破门”“球进了”等，可输出 GOAL；
3. 如果文本中出现“扑出”“门将扑救”“被挡出”等，应优先判断为 SAVE 或 SHOT_BLOCKED；
4. 如果文本中出现“越位”“犯规”“黄牌”“红牌”“点球”“VAR 回看”等词，应结合上下文输出相应事件；
5. 如果文本只体现一般进攻推进而没有明确事件，不要强行输出结果；
6. 时间戳应尽量依据解说句所在时间位置估计；
7. 输出必须严格符合 JSON 结构，不要附加说明文字。

输出格式如下：
{
  "events": [
    {
      "event_type": "FOUL",
      "timestamp_seconds": 512.4,
      "confidence": 0.76,
      "description": "Commentary indicates a foul in midfield"
    }
  ],
  "video_duration": 0
}
```

### 音频链路提示词设计说明

音频语义链路的提示词设计重点与视觉链路不同，它更强调“基于语言内容理解事件”，而不是“基于画面判断事件”。

第一，音频链路提示词需要明确告诉模型输入的是“带时间戳的解说文本”，这样模型才能理解每条句子既包含语义信息，也包含对应的时间位置。

第二，音频链路中对一些判罚类事件的效果通常优于纯视觉链路。因为像“越位”“犯规”“点球”“VAR”这类事件，很多时候画面未必足够清楚，但解说文本会直接给出判断。因此，这一链路更适合作为语义补充，而不是单独替代视觉分析。

第三，音频链路输出的事件结构必须与视觉链路保持一致。只有两条链路输出相同的事件字段，后续的融合、去重和片段规划才容易统一处理。

## 5.3.3 两条链路的区别与协同设计

两条链路虽然都属于大模型调用，但它们在任务定位、输入信息和适用事件类型上并不相同。视觉链路更偏向“看见了什么”，音频语义链路更偏向“听到了什么、解说表达了什么”。因此，这两条链路之间不是重复关系，而是互补关系。

### 两条链路差异对比

| 对比项 | 视觉检测链路 | 音频语义链路 |
|---|---|---|
| 主要模型 | `qwen3-vl-flash` | `qwen3-asr-flash` + `qwen-plus` |
| 输入数据 | 视频或关键帧 | 视频抽取出的音频及转写文本 |
| 主要依据 | 画面证据、动作、场景变化 | 解说语义、语音内容、关键词 |
| 更擅长识别的事件 | 进球、射门、扑救、角球 | 犯规、越位、黄牌、红牌、VAR、点球 |
| 容易受影响的因素 | 镜头切换、画质、视角遮挡 | 解说缺失、语音噪声、文本歧义 |
| 输出形式 | JSON 事件列表 | JSON 事件列表 |

从表中可以看出，两条链路最大的区别不在于模型来自哪一家，而在于它们处理的是不同模态的信息。即使两条链路都使用阿里模型体系，视觉链路和音频链路承担的任务仍然不同：前者负责从画面中寻找事件证据，后者负责从语言和声音中补充语义判断。

### 双链路同时保留的必要性

如果系统只保留视觉链路，那么在某些画面不清晰或镜头切换过快的情况下，像犯规、越位、VAR 这类事件就容易漏掉；如果系统只保留音频语义链路，那么在没有清晰解说、现场语音较弱或者解说滞后的情况下，进球、扑救和射门等事件又可能判断不准。因此，从系统设计角度看，双链路并行的意义在于：

- 让视觉信息和语义信息互相补充；
- 降低单一模态失败带来的整体风险；
- 提高复杂比赛视频下的事件覆盖率；
- 在一条链路失效时，另一条链路仍可提供基础结果。

### 融合与降级策略

系统在运行时按照以下逻辑处理两条链路的输出：

1. 当视觉链路和音频语义链路都成功时，对两者结果进行融合；
2. 如果同类型事件在时间上足够接近，则认为它们描述的是同一比赛事件；
3. 融合时对时间戳、置信度和描述文本进行统一处理；
4. 如果只有一条链路成功，则保留该链路结果继续下游流程；
5. 如果两条链路都失败，则返回检测失败状态。

这种设计保证了系统既能够在理想情况下获得更完整的检测结果，也能够在单链路异常时维持基本功能，体现了原型系统在工程实现上的稳定性考虑。
